# Agent 回答可靠性评估算法说明

本文档说明本项目中 Agent 调度面板里的“综合可靠性”如何计算，以及为什么它不等同于大模型自报置信度。

## 1. 设计背景

本项目是基于 RAG 的医疗健康问答 Agent。医疗场景不能简单展示“模型有 80% 把握”这类数字，因为大模型自报置信度并不稳定，也不能直接代表医学正确率。

因此，本项目将原来的单一 `confidence` 升级为“回答可靠性评估”，从多个可解释维度综合判断本次回答是否有足够依据。

普通用户前端主要展示自然语言提示，例如：

- 资料匹配充分
- 资料匹配一般
- 依据偏弱
- 依据不足

管理员前端展示完整 Agent 调度细节，包括综合可靠性、风险等级和各项评分。

## 3. 综合可靠性公式

综合可靠性由六个维度加权得到：

```text
综合可靠性 =
0.30 * 知识库匹配度
+ 0.25 * 回答支撑度
+ 0.15 * 问题相关度
+ 0.10 * 输入完整度
+ 0.10 * 多轮一致性
+ 0.10 * 来源权威度
```

代码中的权重如下：

```python
weights = {
    "knowledge_match": 0.30,
    "faithfulness": 0.25,
    "answer_relevance": 0.15,
    "input_completeness": 0.10,
    "consistency": 0.10,
    "source_authority": 0.10,
}
```

最终分数范围为 `0.00 - 1.00`，前端显示为百分比。

例如综合可靠性 `0.52`，前端显示为 `52%`。

## 4. 公式来源与权重依据

当前公式不是从某一篇论文中原封不动复制的标准公式，而是结合本项目场景设计的工程型启发式评分模型。

它的来源可以概括为：

```text
RAG 评估思想 + 医疗问答安全需求 + 当前项目已有数据结构 + 可解释性要求
```

其中，RAG 评估思想主要参考了类似 RAGAS 的评价维度，例如：

- Context Precision / Context Relevancy：上下文是否和问题相关；
- Response Relevancy：回答是否回应了用户问题；
- Faithfulness：回答是否能被检索上下文支撑。

但是医疗问答场景还需要额外考虑：

- 用户输入是否完整；
- 是否命中危险症状规则；
- 药品说明书、疾病知识库等来源是否可靠；
- 多轮对话中“这个药”“这个症状”是否被正确继承。

因此，本项目没有直接使用单一模型置信度，而是设计了六个可解释分项。

### 4.1 为什么知识库匹配度权重最高

知识库匹配度权重为 `0.30`。

原因是本项目的核心定位是 RAG 医疗问答 Agent，而不是纯聊天模型。系统回答是否可靠，首先取决于有没有检索到和用户问题相关的知识库内容。

如果知识库匹配很弱，即使大模型语言表达很流畅，也不能说明回答可靠。

### 4.2 为什么回答支撑度权重为 0.25

回答支撑度权重为 `0.25`。

医疗问答最需要避免的是模型凭空生成。回答支撑度用于衡量回答中的主要内容是否能被检索资料支持。

它和知识库匹配度不同：

- 知识库匹配度关注“有没有找到相关资料”；
- 回答支撑度关注“最终回答有没有依据这些资料”。

因此它的权重仅次于知识库匹配度。

### 4.3 为什么问题相关度权重为 0.15

问题相关度权重为 `0.15`。

它主要用于避免答非所问。例如之前出现过用户上传布洛芬图片后继续问“这个药有什么禁忌”，但 Agent 回答成阿司匹林的情况。

这类错误不是知识库不存在，而是 Agent 没有正确理解当前问题对象，所以需要单独评分。

### 4.4 为什么输入完整度权重为 0.10

输入完整度权重为 `0.10`。

医疗问答对上下文信息依赖较强。用户只输入“我不舒服”和用户输入“发热两天、咳嗽、有痰、没有胸痛”时，系统可判断的信息量完全不同。

输入完整度不会直接决定答案正确性，但会影响回答能否具体、可靠。

### 4.5 为什么多轮一致性权重为 0.10

多轮一致性权重为 `0.10`。

Agent 系统必须能正确继承上下文，例如：

```text
用户：上传布洛芬片图片
系统：识别并查询布洛芬片
用户：这个药有什么禁忌？
```

此时“这个药”应该指向“布洛芬片”。如果 Agent 继承错对象，回答就会明显不可靠。

### 4.6 为什么来源权威度权重为 0.10

来源权威度权重为 `0.10`。

医疗健康系统中，不同来源的可信程度不同。例如药品说明书通常比普通网络文本更适合作为用药依据。

当前项目主要区分：

- 药品说明书；
- 疾病知识库；
- 危险规则；
- 无明确来源。

后续可以进一步扩展为临床指南、医院科普、用户上传未审核资料等来源等级。

### 4.7 公式性质说明

该公式是一个可解释的初始工程评分模型，不是最终医学准确率模型。

它的优点是：

- 不额外调用模型，响应速度快；
- 每个分数都有明确含义；
- 普通用户不会被数字误导；
- 管理员可以看到 Agent 哪一环出了问题；
- 答辩时能够解释系统为什么认为某次回答可靠或不可靠。

它的后续优化方向是根据真实测试集、用户反馈和管理员标注数据继续校准权重。

## 5. 六个评分维度说明

### 5.1 知识库匹配度 knowledge_match

表示 RAG 或药品知识库是否检索到了与问题相关的资料。

当前逻辑：

- 危险提醒命中规则：较高
- 药品查询命中药品库：较高
- RAG 检索最高分较高：较高
- RAG 检索结果弱或为空：较低

它回答的问题是：

```text
系统有没有找到可参考的知识库依据？
```

### 5.2 回答支撑度 faithfulness

表示回答内容是否能被检索到的知识片段支持。

当前实现使用轻量文本重叠方法进行近似判断：

- 将回答文本拆分为关键词
- 将检索文档内容拆分为关键词
- 判断回答里的关键信息有多少能在资料中找到对应

它回答的问题是：

```text
回答是不是主要依据知识库生成，而不是凭空发挥？
```

### 5.3 问题相关度 answer_relevance

表示回答是否真正回应了用户当前问题。

例如用户问：

```text
这个药有什么禁忌？
```

如果上一轮主题是“布洛芬片”，但回答跳到了“阿司匹林”，问题相关度就应该降低。

它回答的问题是：

```text
回答有没有答到用户真正问的对象和问题？
```

### 5.4 输入完整度 input_completeness

表示用户输入信息是否足够。

当前会考虑：

- 文本长度
- 是否包含症状持续时间
- 是否包含疼痛、发热、瘙痒、禁忌、剂量等关键线索
- 是否上传图片
- 是否有图片识别标签
- 药品查询是否识别出明确药名

它回答的问题是：

```text
用户提供的信息够不够支持一次较完整的判断？
```

### 5.5 多轮一致性 consistency

表示当前回答是否和多轮对话上下文保持一致。

例如：

- 上一轮图片识别为“布洛芬片”
- 用户下一轮问“这个药有什么禁忌”
- Agent 应继续围绕“布洛芬片”回答

如果当前主题和检索文档标题不一致，或回答没有围绕当前主题，多轮一致性会降低。

它回答的问题是：

```text
Agent 有没有记住当前正在讨论的对象？
```

### 5.6 来源权威度 source_authority

表示本次回答依据的来源类型是否可靠。

当前项目中：

- 药品说明书类数据权重较高
- 疾病知识库数据权重较高
- 无检索来源则较低

它回答的问题是：

```text
回答依据是不是来自项目内可信知识源？
```

## 6. 等级划分

综合可靠性会映射成自然语言等级：

```text
>= 0.80: high         资料匹配充分
>= 0.60: medium       资料匹配一般
>= 0.40: low          依据偏弱
<  0.40: insufficient 依据不足
```

对应代码：

```python
def reliability_level(score):
    if score >= 0.8:
        return "high"
    if score >= 0.6:
        return "medium"
    if score >= 0.4:
        return "low"
    return "insufficient"
```

## 7. 截图中的示例解释

截图中显示：

```text
综合可靠性：52%
安全风险：中
知识库匹配：48%
回答支撑度：53%
问题相关度：45%
输入完整度：50%
多轮一致性：45%
来源权威度：82%
```

可以理解为：

- 来源权威度较高，说明检索到的资料来自项目知识库；
- 但知识库匹配度只有 48%，说明检索到的资料和用户问题并不够强相关；
- 问题相关度和多轮一致性都只有 45%，说明 Agent 可能没有很好地围绕当前问题或上下文回答；
- 输入完整度 50%，说明用户输入信息偏少；
- 最终综合后得到 52%，属于“依据偏弱/资料匹配一般”的边界状态。

这类情况不应该让普通用户看到“模型置信度 52%”，而应该提示：

```text
当前依据偏弱，建议补充更多症状、持续时间、图片或药品信息。
```

## 8. 为什么它不是模型概率

本项目中的综合可靠性不是：

```text
模型认为自己有多少概率回答正确
```

而是：

```text
系统根据检索结果、回答内容、输入完整度、上下文一致性和来源类型计算出的可解释评分
```

原因是：

- 大模型自报置信度不稳定；
- RAG 检索分数高不代表最终回答一定正确；
- 医疗问答需要考虑安全风险和信息完整度；
- 项目答辩时需要能解释每个分数来源。

因此，本项目采用可解释的本地综合评分，而不是直接相信大模型给出的置信度。

## 9. 当前算法局限

当前版本是轻量级本地评分，不额外调用评估模型，因此响应速度较快，但也有局限：

- 回答支撑度使用关键词重叠近似，不是真正的语义蕴含判断；
- 问题相关度没有使用专业 reranker；
- 多轮一致性目前主要依赖当前主题和检索标题；
- 来源权威度基于项目内数据类型，尚未区分数据来源等级、更新时间和审核状态；
- 没有引入人工标注数据做校准。

## 10. 后续升级方向

后续可以继续升级为更专业的医疗 RAG 评估体系：

1. 引入 RAGAS 类指标

   - Context Precision
   - Context Recall
   - Response Relevancy
   - Faithfulness

2. 引入大模型评审器

   让文本大模型根据“问题 + 检索资料 + 回答”判断：

   - 回答是否被资料支持
   - 是否答非所问
   - 是否存在医学风险

3. 引入 reranker

   对 RAG 检索结果重新排序，提高知识库匹配度。

4. 引入人工反馈校准

   根据用户反馈、管理员标注和老师评审样本，调整各项权重。

5. 引入知识库来源等级

   给不同来源设置不同权威度，例如：

   - 药品说明书
   - 临床指南
   - 医院科普
   - 普通百科
   - 用户上传未审核资料
